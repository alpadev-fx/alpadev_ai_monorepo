**Date**: 2026-03-28

## Executive Summary

The codebase has excellent architecture (repository-service-router pattern consistently applied, Zod validation on inputs, tRPC type inference) but suffers from **critical security vulnerabilities** that require immediate action. Production secrets are committed to git in `.env.production`, database credentials are hardcoded in source code, the WebSocket server has zero authentication, and IDOR vulnerabilities exist across 5 domains. The test infrastructure is nonexistent (no test runner installed, 0% coverage, CI deploys without any quality gates). The Three.js components have significant memory leaks. The seed system is completely broken from a prior schema migration.

**Immediate action required**: Rotate ALL production secrets (they are burned). Fix IDOR vulnerabilities before next deployment. Add WebSocket authentication.

## P0 -- CRITICAL (Fix Immediately)

### [BH-001] Production secrets committed to git in `.env.production`

- **Found by**: Security Reviewer
- **File**: `.env.production` (entire file)
- **Category**: Security -- Data Exposure
- **Description**: The file `.env.production` is committed to git and contains every production secret in plaintext: MongoDB Atlas credentials, NEXTAUTH_SECRET, Google OAuth client ID/secret, Google refresh token, Gemini API key, GitHub PAT, Resend API key, Cloudflare R2 access key/secret, Mistral API key, DeepSeek API key.
- **Impact**: Total compromise of every external service, database, and authentication mechanism. Anyone with read access to the repo can impersonate any user, access the production database, run up AI API bills, and read/write/delete all R2 storage.
- **Fix**:
  1. **IMMEDIATELY rotate ALL secrets** listed in `.env.production`. Every single one is burned.
  2. Delete `.env.production` from git history using `git filter-repo` or BFG Repo Cleaner.
  3. Verify `.env.production` is in `.gitignore` (`.env` is, but `.env.production` must be explicitly listed).
  4. Use GCP Secret Manager (already partially integrated via `start.sh`) for all production values.

### [BH-002] Hardcoded MongoDB production credentials in source code

- **Found by**: Security Reviewer, Architect
- **File**: `packages/db/db.ts:75`
- **Category**: Security -- Data Exposure
- **Description**: A Railway MongoDB connection string with full username and password is hardcoded as a fallback: `mongodb://mongo:hvrnIjFrwgEDQhUxYPYulIEPtWZmgxok@centerbeam.proxy.rlwy.net:16709/alpadev`. If `MONGO_URL` is ever unset in production, the code silently falls back to this.
- **Impact**: Full read/write access to the Railway MongoDB instance for anyone reading source code.
- **Fix**:
  ```typescript
  // BEFORE (db.ts:73-77)
  if (process.env.NODE_ENV === 'production') {
    console.error("[db] Using fallback URL - THIS WILL LIKELY FAIL!")
    const fallbackMongoUrl = "mongodb://mongo:hvrnIjFrwgEDQhUxYPYulIEPtWZmgxok@centerbeam.proxy.rlwy.net:16709/alpadev?authSource=admin"
    process.env.MONGO_URL = fallbackMongoUrl
    return fallbackMongoUrl
  }

  // AFTER
  if (process.env.NODE_ENV === 'production') {
    throw new Error("CRITICAL: No database URL found. Set MONGO_URL environment variable.")
  }
  ```

### [BH-003] NEXTAUTH_SECRET fallback enables JWT forgery

- **Found by**: Security Reviewer
- **File**: `packages/auth/src/auth/authOptions.ts:118`
- **Category**: Security -- Auth
- **Description**: `secret: process.env.NEXTAUTH_SECRET || "fallback-secret-for-dev"` means if the env var is missing at runtime, the app uses a publicly known string. Any attacker can forge valid JWTs and impersonate any user, including admins.
- **Impact**: Complete authentication bypass, admin impersonation.
- **Fix**:
  ```typescript
  // BEFORE
  secret: process.env.NEXTAUTH_SECRET || "fallback-secret-for-dev",

  // AFTER
  secret: (() => {
    const secret = process.env.NEXTAUTH_SECRET;
    if (!secret) throw new Error("NEXTAUTH_SECRET environment variable is required");
    return secret;
  })(),
  ```

### [BH-020] Hyperspeed Three.js scene: window resize listener permanently leaked + GPU resources never disposed

- **Found by**: Performance Engineer
- **File**: `apps/frontend/components/shared/Hyperspeed.tsx:1068,1232-1256`
- **Category**: Performance -- Memory Leak
- **Description**: Two compounding issues: (1) `.bind(this)` in addEventListener creates a different reference than `.bind(this)` in removeEventListener -- the listener is never removed. (2) `scene.clear()` only removes objects from scene graph, doesn't call `.dispose()` on geometries, materials, textures.
- **Impact**: Each mount/unmount leaks the entire Three.js scene (~10-30MB). Navigating away and back 5 times = 50-150MB leaked GPU memory. Can trigger WebGL context loss on mobile.
- **Fix**: Store bound reference in constructor, traverse scene and dispose all GPU resources before clearing.

## Systemic Recommendations

### 1. Security-First Sprint
The density of IDOR vulnerabilities (5 domains affected) suggests a systemic gap: **ownership checks are not enforced by convention**. Create a `withOwnership` middleware or repository wrapper that automatically scopes queries by `userId` from the session context.

### 2. Build Quality Gates
The combination of `ignoreBuildErrors: true` + no CI tests is the root cause of many bugs surviving to production. Fix this first -- it's the multiplier that makes every other bug harder to catch.

### 3. Test Foundation
Start with pure-function tests (zero dependencies): `forecast()`, `kpi()`, `simulate()`, `MessageAnalysisService.analyze()`, import header mapping. These 5 test files would cover the highest-risk business logic with minimal setup.

### 4. Three.js Resource Management
Create a `useThreeCleanup` hook that automatically disposes geometries, materials, and textures on unmount. Apply to all Canvas components. Add IntersectionObserver to all shader components for visibility-based rendering.

### 5. Schema Alignment Audit
The request domain has drifted significantly from its Prisma schema (5 phantom fields). Run a systematic check of all Zod schemas against their Prisma models to find similar drift in other domains.
